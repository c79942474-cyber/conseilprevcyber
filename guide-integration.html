1Vue d'ensemble & modèle de sécurité

Le raccordement repose sur un connecteur : un programme Python léger, fourni par CONSEILPREV, que vous exécutez dans votre environnement (collecteur, VM, poste d'administration). Il lit votre source de données puis pousse des événements normalisés vers le cockpit.

[ Plateforme OT / syslog / CSV ]  →  [ Connecteur (chez vous) ]  →  HTTPS sortant  →  [ Cockpit /api/ingest ]

Principes de sécurité

  • Flux sortant uniquement : le connecteur ouvre une connexion HTTPS vers le cockpit. Aucun flux entrant n'est requis vers votre réseau industriel.
  • Collecte passive : lecture d'API (GET) ou réception syslog — le connecteur n'émet rien vers vos automates et n'écrit rien sur votre plateforme.
  • Authentification par jeton : chaque envoi porte l'en-tête X-Ingest-Token ; sans jeton valide, le cockpit refuse (401).
  • Cohérence IEC 62443 : placez le connecteur dans une zone de supervision dédiée ; le flux sortant vers le cockpit constitue un conduit maîtrisé et documenté.
Où l'installer ? Sur une machine qui ① voit l'API de votre plateforme (ou reçoit ses syslog) et ② peut sortir en HTTPS (port 443) vers le cockpit. Typiquement : une VM de la zone de supervision ou une DMZ industrielle.

2Prérequis

ÉlémentDétail
Machine hôteLinux, Windows ou macOS — 1 vCPU / 512 Mo suffisent. Pour la production : Linux avec systemd recommandé.
PythonPython 3.8 ou plus (vérifier : python3 --version). Aucune dépendance à installer (bibliothèque standard uniquement).
Flux réseauSortant HTTPS (443) vers le cockpit. Selon la source : accès à l'API de la plateforme, ou port UDP local (ex. 5514) pour recevoir le syslog.
Jeton d'ingestionINGEST_TOKEN — remis par l'administrateur du cockpit (visible pour lui sur la page Connecter).
Accès plateformeUn jeton/compte API en lecture seule sur votre plateforme OT (ou une règle de transfert syslog/CEF).
À demander à votre équipe réseau : l'ouverture du flux sortant « connecteur → cockpit » en HTTPS, et si besoin le renvoi syslog « plateforme → connecteur ». Rien d'autre.

3Installation

  1. Téléchargez le connecteur : conseilprev-connecteur.zip (bouton aussi disponible sur la page Connecter).
  2. Décompressez-le sur la machine hôte :
mkdir -p /opt/conseilprev && cd /opt/conseilprev
unzip ~/conseilprev-connecteur.zip
# le dossier connectors/ contient : connector.py (CLI), les sources (API, syslog, CSV),
# un mock de test, et deploy/ (guide + service systemd + Dockerfile)
  1. Vérifiez que tout répond :
python3 -m connectors.connector --help
python3 -m connectors.connector presets      # liste des plateformes préconfigurées
Aucun pip install, aucun accès Internet requis pour l'installation : le connecteur n'utilise que la bibliothèque standard Python.

4Configuration & premier test

4.1 Variables d'environnement

export COCKPIT_URL="{ORIGIN}"
export INGEST_TOKEN="VOTRE_JETON"          # remis par l'administrateur
# PKI interne (optionnel) : export COCKPIT_CAFILE=/chemin/ca.pem

4.2 Test à blanc (aucun envoi)

Le mode --dry-run affiche ce qui serait envoyé, sans rien transmettre — idéal pour valider le mapping avant d'ouvrir le flux :

python3 -m connectors.connector demo --dry-run

4.3 Premier envoi réel

python3 -m connectors.connector demo --interval 1
# puis ouvrez le cockpit → « ◉ Temps réel » : les événements de démonstration apparaissent

Ou en un seul appel HTTP (sans le connecteur) :

curl -X POST {ORIGIN}/api/ingest \
  -H "content-type: application/json" -H "X-Ingest-Token: $INGEST_TOKEN" \
  -d '{"asset":"Automate-Test","zone":"Supervision (SCADA)","type":"discovery","event":"Premier événement","severity":"info"}'

5Selon votre plateforme

Les presets préconfigurent l'endpoint attendu et la correspondance des champs. Deux modes selon la plateforme :

ModePrincipeQuand le choisir
Pull (API)Le connecteur interroge l'API de la plateforme toutes les N secondes (lecture seule, dédoublonnage automatique).Vous avez un compte API lecture seule.
Push (syslog/CEF)La plateforme émet ses alertes vers le connecteur qui écoute en UDP.Le plus simple si la plateforme sait « forwarder » (Defender for IoT, Nozomi, Claroty CTD, Forescout…).

Nozomi Networks (Guardian / Vantage)

# Actifs OT — le score de risque (0-100) pondère les zones du cockpit
python3 -m connectors.connector otplatform --preset nozomi_assets \
  --url "https://<guardian>/api/open/query/do?query=nodes" \
  --header "Authorization: Bearer $TOKEN_PLATEFORME" --interval 30
# Alertes de sécurité
python3 -m connectors.connector otplatform --preset nozomi \
  --url "https://<guardian>/api/open/query/do?query=alerts" \
  --header "Authorization: Bearer $TOKEN_PLATEFORME" --interval 10
# Réseaux sans fil (inventaire Wi-Fi/IIoT)
python3 -m connectors.connector otplatform --preset nozomi_wireless \
  --url "https://<guardian>/api/open/query/do?query=wireless_networks" \
  --set type=discovery --set "zone=Terrain (capteurs)" --interval 60

Claroty (xDome / CTD)

# Actifs (risk score)
python3 -m connectors.connector otplatform --preset claroty_assets \
  --url "https://<tenant>.claroty.com/api/v1/devices" \

# Alertes
python3 -m connectors.connector otplatform --preset claroty \
  --url "https://<tenant>.claroty.com/api/v1/alerts" \

# CTD on-prem : possible aussi en transfert syslog/CEF → voir « Syslog / CEF »

Tenable OT Security

# Actifs (risque 0-100 ; ≥ 70 = high risk chez Tenable, aligné sur le cockpit)
python3 -m connectors.connector otplatform --preset tenable_ot_assets \

python3 -m connectors.connector otplatform --preset tenable_ot \

# Recommandé : transfert syslog/CEF du capteur vers le connecteur

python3 -m connectors.connector otplatform --preset defender_iot_sensor \
  --url "https://<capteur>/api/v1/alerts" --header "Authorization: Bearer $TOKEN_PLATEFORME" --interval 15

Syslog / CEF (toute source)

# Écoute UDP — pointez firewalls, switches, sondes vers le connecteur (port 5514)
python3 -m connectors.connector syslog --listen 0.0.0.0:5514
# Ou suivi d'un fichier de log
python3 -m connectors.connector syslog --file /var/log/syslog

Le format CEF est décodé automatiquement (nom d'alerte, sévérité, hôte, zone).

Export CSV

python3 -m connectors.connector csv --file mon-export.csv --interval 1
# Colonnes reconnues (toutes optionnelles) : asset, zone, type, event, severity, ts
Champ différent dans votre version ? Chaque champ se surcharge sans toucher au code : --map severity=risk_level, --map asset=hostname… Et --set champ=valeur force une constante (ex. --set "zone=Terrain (capteurs)"). La liste des mappings par preset : python3 -m connectors.connector presets.

6Modèle de données (référence)

Chaque événement envoyé au cockpit est normalisé vers :

{ "asset": "Automate S7-1500", "zone": "Supervision (SCADA)",
  "type": "discovery", "event": "Nouvel actif inventorié",
  "severity": "info", "ts": 1712345678901 }
ChampRôleValeurs
assetNom de l'équipement (alimente l'inventaire — dédoublonné)libre (120 car. max)
zoneRattachement à l'une des 5 zones du cockpitEntreprise (IT) · DMZ industrielle · Supervision (SCADA) · Cellule / Contrôle · Terrain (capteurs) — rattachement automatique par mots-clés, sinon --set zone=…
typeNature de l'événementdiscovery (découverte), patch (correctif), autre = alerte/info
eventTexte affiché dans le journallibre (240 car. max)
severityGravitéTexte : critical/high/major → critique ; medium/minor/warning → avertissement ; low/info → info. Numérique (score de risque) : 0-10 (≥7 critique, 4-6 avert.) ou 0-100 (≥70 critique, 40-69 avert.)
tsHorodatage (ms epoch)ajouté automatiquement si absent

7Mise en production 24/7 (systemd)

  1. Créez le fichier d'environnement (contient les secrets — protégez-le) :
sudo mkdir -p /etc/conseilprev
sudo cp connectors/deploy/connector.env.example /etc/conseilprev/connector.env
sudo nano /etc/conseilprev/connector.env      # COCKPIT_URL, INGEST_TOKEN, OT_API_URL, OT_API_TOKEN, OT_PRESET…
sudo chmod 600 /etc/conseilprev/connector.env
  1. Installez et démarrez le service :
sudo cp connectors/deploy/conseilprev-connector.service /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl enable --now conseilprev-connector
journalctl -u conseilprev-connector -f          # suivre les logs

Le service redémarre automatiquement en cas de coupure et au démarrage de la machine. Une alternative Docker est fournie (connectors/deploy/Dockerfile).

8Supervision du connecteur

Ajoutez --metrics-port 9109 à n'importe quelle commande : le connecteur expose alors

  • /metrics — métriques Prometheus : événements envoyés/échoués, horodatage du dernier succès, disponibilité ;
  • /healthz — sonde de vie (supervision simple, load-balancer, Zabbix…).
scrape_configs:
  - job_name: conseilprev-connector
    static_configs: [{ targets: ['collecteur.interne:9109'] }]

Alertes recommandées : scrape en échec (connecteur mort) · aucun envoi réussi depuis 15 min · compteur d'échecs qui augmente. Exemples de règles complets dans connectors/deploy/DEPLOY.md (§10).

9Recette de l'intégration (checklist)

  • connector … --dry-run affiche des événements corrects (bonnes zones, bonnes sévérités).
  • Le test curl renvoie {"ok": true}.
  • Le cockpit en mode ◉ Temps réel affiche les événements (journal + inventaire).
  • Les zones passent en surveillance/alerte selon la gravité attendue.
  • La page Tendances se construit (historique persisté).
  • Le service systemd est active (running) et survit à un reboot.
  • Les métriques répondent sur :9109/metrics et l'alerte « connecteur mort » est en place.
  • Le jeton plateforme est en lecture seule ; le fichier d'env est en chmod 600.

10Dépannage

SymptômeCause probableCorrectif
« AUTH 401 » côté connecteurJeton d'ingestion absent ou erronéVérifiez INGEST_TOKEN (identique à celui du cockpit — demandez à l'administrateur).
« 503 ingestion désactivée »Le cockpit n'a pas de jeton configuré côté serveurL'administrateur doit définir INGEST_TOKEN dans l'hébergement du cockpit.
Erreur TLS / certificatPKI interne, proxy SSL--cafile /chemin/ca.pem (bundle de votre PKI). Ne désactivez jamais la vérification en production.
« Plateforme injoignable »URL/API inaccessibles depuis la machine hôteTestez curl -k https://plateforme/… depuis la même machine ; vérifiez pare-feu et DNS.
Rien ne s'affiche en Temps réelÉvénements envoyés avant l'ouverture de la page ?Le cockpit charge l'état courant à la connexion — vérifiez /api/state (compte requis) et le journal du connecteur (« OK … »).
Tous les événements en « info »Champ de gravité non mappéSurchargez : --map severity=risk (ou le nom exact du champ chez vous).
Zone « — » ou mal rattachéeLibellé de zone inconnu du cockpitForcez : --set "zone=Cellule / Contrôle" (une des 5 zones), ou mappez : --map zone=site_name.
Doublons d'événementsChamp identifiant absentPrécisez --id-field (défaut id) — le dédoublonnage s'appuie dessus.

11Bonnes pratiques sécurité

  • Secrets : jeton d'ingestion et jeton plateforme uniquement en variables d'environnement (chmod 600), jamais dans un dépôt ni un ticket. Prévoyez une rotation périodique.
  • Moindre privilège : compte API plateforme en lecture seule ; le connecteur n'a besoin de rien d'autre.
  • Cloisonnement : connecteur en zone de supervision ; seul le flux sortant HTTPS est ouvert (conduit documenté, cohérent IEC 62443).
  • Confidentialité : n'envoyez de vraies données que vers une instance de cockpit privée (comptes utilisateurs activés — c'est le cas ici).
  • Traçabilité : conservez les journaux du connecteur (systemd/journald) et surveillez les métriques (§8).
